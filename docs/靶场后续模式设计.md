# 靶场后续模式设计

本文记录《我的方块靶场》后续从单一靶场 demo 扩展为“武器试验场 + 地图射击模式”的产品与架构方向。它不是立即重构清单，而是后续做功能拆分、排查设计取舍和制定实施计划时的基准文档。

最后整理日期：2026-07-11。

## 总体方向

项目后续建议拆成三类模式：

- **当前靶场模式**：保留现有方块靶场/防守体验，作为稳定可试玩入口。
- **武器试验场**：把武器系统、后坐力、枪口火焰、命中、DPS 和靶标行为做成可调试、可观察、可复现的实验场。
- **动态地图模式**：单独做开放地图射击玩法，可基于 Babylon Dynamic Terrain 做地形原型，不替代当前靶场。

核心原则是：武器、命中、目标、UI、调试探针尽量复用；场景环境、刷怪方式、胜负目标和玩法循环按模式隔离。

## 模式边界

### 当前靶场模式

定位：稳定试玩和基础玩法验证。

保留内容：

- 现有 5 把第一人称 3D 武器。
- 当前敌人、命中、计分、倒计时、基地血量和 HUD。
- Playwright 截图验收、3D 武器可见性验收、枪口火焰锚点验收。

不建议在这里直接塞入太多实验功能。当前模式要继续承担“能稳定玩、能快速验证回归”的职责。

### 武器试验场

定位：武器机制和手感的实验室，优先级最高。

建议能力：

- **无敌人模式**：只保留武器、墙面、弹孔和数据面板，用来测后坐力、散布和枪口火焰。
- **死靶模式**：静态靶或假人，用来测试单发伤害、距离衰减、爆头/身体命中和换弹节奏。
- **动靶模式**：沿固定路线移动的靶，用来练跟枪和验证命中判定。
- **敌人模式**：复用当前敌人逻辑，用来做轻量实战模拟。
- **弹道墙/涂鸦墙**：用 Babylon Decal 或轻量贴花/标记记录弹孔，让连续射击的后坐力轨迹可视化。
- **数据看板**：显示当前武器、弹匣、射速、命中数、命中率、DPS、爆头率、总伤害和最近一轮射击统计。

验收重点：

- 长按开火时，弹孔轨迹能真实反映当前武器 recoil/cameraKick。
- 切换武器后数据重置或分组清晰，不混淆统计。
- 调参数后能快速看出后坐力、散布、枪口火焰和命中反馈变化。
- 不依赖后端，优先用浏览器内状态和截图完成调试。

### 动态地图模式

定位：独立的开放地图射击实验，不和当前靶场强行合并。

可参考 Babylon Dynamic Terrain 官方入口：

- <https://doc.babylonjs.com/communityExtensions/dynamicTerrains/>

建议能力：

- 使用动态地形生成可移动的大范围地图。
- 以玩家为中心，在一定距离的圆环区域刷怪或刷靶。
- 怪物从固定 lane 改为朝玩家位置追踪。
- 只对玩家附近区域做必要的高度查询、碰撞和对象更新。
- 保持方块/像素审美，避免突然变成写实风。

地形风格有两条路线：

- **方块阶梯地形**：更贴近当前 Minecraft 风格，碰撞和视觉统一性更好，但地形生成需要做阶梯化处理。
- **低多边形平滑地形**：实现更快，适合先做原型，但要用像素贴图、低多边形材质和物件风格把画面拉回项目调性。

建议先做低风险原型：只验证相机移动、地形生成、高度采样、基本碰撞和性能，再决定是否做方块阶梯化。

## 架构建议

后续不要一次性把 `src/main.js` 大重构成复杂框架，建议从轻量模式生命周期开始。

建议抽象：

- **Mode Manager**：负责进入/退出模式、清理场景对象、切换 HUD 和注册输入。
- **共享武器系统**：继续复用当前武器配置、射击、换弹、枪口火焰锚点和 Playwright 调试探针。
- **共享命中系统**：把射线、命中点、伤害结算、弹孔/命中特效做成可被不同模式调用的能力。
- **目标/敌人生成器**：靶场用固定点位和 lane，大地图用玩家周围圆环刷新。
- **模式级 UI**：不同模式有自己的面板，但底层 HUD 控件和状态更新接口尽量复用。

推荐状态结构：

```text
app
  scene / engine / camera
  assets
  activeMode
  sharedSystems
    weapons
    combat
    effects
    input
    ui
  modes
    rangeMode
    weaponLabMode
    terrainMode
```

模式切换要注意：

- 清理上一模式创建的 mesh、材质、粒子、敌人、计时器和事件监听。
- 不重复加载同一批资源。
- 输入状态要重置，避免切模式后鼠标按住、换弹、连发状态残留。
- E2E 调试探针要能指定进入某个模式，而不是只能进入默认靶场。

## 实施顺序

### 阶段 1：武器试验场最小可用版

目标：先让武器系统变成可观察的调试工具。

建议范围：

- 增加模式入口或调试参数，例如进入 `weaponLabMode`。
- 建一个简单封闭试验空间：地面、弹道墙、基础灯光。
- 支持切换无敌人、死靶、动靶、敌人四种测试对象。
- 增加弹孔记录和射击统计。
- 保留当前 3D 武器、枪口火焰和 HUD 验收。

#### 实施进度（2026-07-11）

武器试验场 MVP 已落地，实际接入点：

#### 修复批次（2026-07-11）

基于只读审查修复 5 个阻断"手动验收通过"的问题，并补 weaponLab 专属 E2E：

## 测试与验收

每个新模式都要同时保留自动测试和浏览器验收。

基础验收：

- `npm run test`
- `npm run build`
- 视觉或玩法改动后运行 `npm run test:e2e`

武器试验场验收：

- 每把武器能开火、换弹、扣弹和显示枪口火焰。
- 弹孔位置和射线命中点一致。
- 长按开火能看出不同武器的后坐力轨迹。
- 数据看板不会影响射击主流程。

动态地图验收：

- 进入地图后画布不空白，地形可见。
- 相机移动时高度合理，不穿地或突然弹飞。
- 玩家附近地形和目标更新正常，远处对象不会造成明显卡顿。
- 控制台没有模型、贴图、地形插件相关错误。
