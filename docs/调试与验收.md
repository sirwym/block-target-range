# 调试与验收

本文记录当前靶场项目的调试入口、浏览器验收清单和常见故障排查方向。

## 调试参数

在本地开发服务器 URL 后追加参数即可启用调试视图，例如 `/?debugWeapon=1`。

- `debugHitbox=1`：显示或辅助检查命中盒。
- `debugActor=1`：显示敌人结构、贴图或行为相关调试信息。
- `debugWeapon=1`：检查当前武器显示、P90 模型状态和第一人称武器表现。
- `debugWeapon2D=1`：检查 2D 武器朝向、缩放、镜像和位置。

## 浏览器验收清单

玩法、资源、音频、模型或 UI 改动后，建议至少完成以下检查：

- 页面正常进入游戏，控制台没有红色错误。
- `Digit1` 到 `Digit5` 可以切换 Glock 17、M4、AK47、AWP、P90。
- 每把枪可以射击、扣弹、换弹，HUD 弹药显示正确。
- 自动武器按住鼠标能持续射击，不会几发后卡住。
- 命中敌人后有命中反馈、血条、飘字、得分和连击反馈。
- 2D 枪口朝向敌人，枪把位于玩家右下角附近。
- 2D 枪口方向以真实浏览器截图为准，不能只靠像素算法或配置快照判断；`flipY` 变更尤其要确认没有把枪口压向热栏。
- P90 3D 模型可见时不遮挡准星；加载失败时回退 2D 图标。
- Network 中 OGG、模型和贴图请求数量合理，没有几分钟几万条重复请求。
- 结束一局后结算面板、重新开始、音效和控制状态正常。

## 测试优先级

测试不是最后补流程，而是本项目防止玩法回归的主要护栏。修改武器、音频、资源、命中、碰撞、模型或 UI 状态时，先判断是否需要补测试，再开始验收。

- 自动测试优先覆盖稳定规则：武器状态、弹匣和换弹、自动/半自动射击、音频缓存、资源存在性、碰撞、评分、P90 模型结构。
- 浏览器验收优先覆盖真实体验：持续射击、鼠标按住连发、控制台错误、Network 请求数量、2D 武器朝向、P90 可见性和 HUD 状态。
- 只跑自动测试不能替代浏览器验收；枪模朝向、模型遮挡、音频请求暴涨这类问题必须在真实页面里看。
- 如果修复线上或手动复现过的问题，测试里要尽量留下对应的回归用例，避免以后重复踩坑。

## 常见故障排查

### 命中后卡住

优先检查控制台是否出现命中反馈相关错误。飘字、特效、HUD 更新不能中断射击主链路；即使视觉反馈失败，也要保证扣血、扣弹、计分和主循环继续运行。

### 音频请求暴涨

检查 `src/audio.js` 的缓存状态。每个 OGG 应该最多加载一次；加载失败后也要记录失败状态，后续播放 fallback，不能反复 fetch 同一路径。

### 2D 武器方向错误

先在 `debugWeapon2D` 中确认图标本身朝向，再校准镜像、旋转、偏移和缩放。不要只靠感觉改一个武器，要同屏对比所有武器。

### P90 模型异常

先区分 glTF 加载失败、Blockbench JSON fallback、贴图缺失和模型坐标问题。模型失败时必须回退 2D，不允许影响其他武器。

### 构建或测试失败

先运行 `npm run test` 定位逻辑测试，再运行 `npm run build` 定位构建或资源路径问题。完整提交前使用 `npm run check`。
