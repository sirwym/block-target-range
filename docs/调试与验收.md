# 调试与验收

本文记录当前靶场项目的调试入口、浏览器验收清单和常见故障排查方向。

## 调试参数

在本地开发服务器 URL 后追加参数即可启用调试视图，例如 `/?debugWeapon=1`。

- `debugHitbox=1`：显示或辅助检查命中盒。
- `debugActor=1`：显示敌人结构、贴图或行为相关调试信息。
- `debugWeapon=1`：检查当前武器显示、TaCZ 模型状态和第一人称武器表现。
- `e2e=1`：启用 Playwright 自动验收探针 `window.__blockTargetRangeDebug`，可启动游戏、切换武器、开火、换弹、临时写入 `modelConfig`，并读取 3D 模型屏幕投影和枪口锚点数据。
- `mode=weaponLab`：进入武器试验场模式。无敌人/倒计时/基地血量，只有地面 + 弹道墙 + 武器 + 数据看板。可配合 `e2e=1` 使用（`?e2e=1&mode=weaponLab`），`snapshot().weaponLab` 暴露弹孔数和双层统计。
- `taczStatic=1`：Phase2 TaCZ 纯静态枪模验收模式。通常与 `?e2e=1&mode=weaponLab&taczStatic=1` 配合使用，只显示裸枪模型，不创建方块手、动画控制器、枪口火焰或换弹姿态。该模式用于判断静态 geo/可见性/静态 pose，不用于判断 ADS、开火、换弹或普通运行时枪口锚点。

## 浏览器验收清单

玩法、资源、音频、模型或 UI 改动后，建议至少完成以下检查：

- 页面正常进入游戏，控制台没有红色错误。
- `Digit1` 到 `Digit5` 可以切换 5 把武器（M4、M95、Deagle Golden、AWP、AK47）。
- 5 把武器都显示 3D 模型，没有第一人称 2D weapon plane。
- 每把枪可以射击、扣弹、换弹，HUD 弹药显示正确。
- 自动武器按住鼠标能持续射击，不会几发后卡住。
- 枪口火焰位于枪管前端，开火、后坐力和换弹时跟随 3D 武器移动。
- 命中敌人后有命中反馈、血条、飘字、得分和连击反馈。
- 3D 武器验收不能只看 mesh 数量。必须确认模型在屏幕右下角有足够投影面积、枪形可辨、无 2D fallback；“mesh 已加载但主画面几乎看不到”视为未修复。
- Network 中 OGG、模型和贴图请求数量合理，没有几分钟几万条重复请求。
- 结束一局后结算面板、重新开始、音效和控制状态正常。

## Phase2 TaCZ 静态验收

Phase2 静态验收与普通运行时验收分离：

- `e2e/phase2-tacz-static.spec.js`：只覆盖 `m4/m95/deagle_golden/awp/ak47` 五把 TaCZ 纯静态枪模。入口固定为 `/?e2e=1&mode=weaponLab&taczStatic=1`，断言 `pureTaczStatic=true`、模型 ready、静态 pose 字段、无手臂/动画/枪火、主结构屏幕诊断不过近裁面。
- `e2e/weapons.spec.js`：继续覆盖普通运行时武器可见性、射击、枪口火焰和换弹状态。运行时姿态或枪口锚点失败属于 Phase3/运行时问题，不直接阻塞 Phase2 静态通过。
- 当前 Phase2 五枪列表与 WEAPON_ORDER 一致，旧专项测试会在运行列表不包含对应武器时跳过，避免把范围外武器 timeout 混入 Phase2 静态验收。

运行 Phase2 静态验收：

```bash
npm run test:e2e -- e2e/phase2-tacz-static.spec.js
```

输出文件：

- `test-results/phase2-static-{id}.png`：每把枪的纯静态截图。
- `test-results/phase2-static-contact-sheet.png`：五枪拼图，交给多模态模型按 `docs/视觉验收清单.md` 复核。
- `test-results/phase2-static-pose-report.json`：姿态搜索报告，包含每把枪的当前 screen bounds、screen outliers 和 top pose candidates。

新增或替换 TaCZ 静态枪时，优先使用浏览器探针：

```js
window.__blockTargetRangeDebug.searchPhase2StaticPose("m95")
window.__blockTargetRangeDebug.searchPhase2StaticPoses()
window.__blockTargetRangeDebug.snapshot().activeModel.debugGeometry.screenOutliers
```

姿态搜索报告只提供候选，不自动写入 `PHASE2_STATIC_POSE_CALIBRATION`。写入前必须确认候选能减少主结构贴近 near plane、保持主枪体连续，并经过多模态截图复验。

## 测试优先级

测试不是最后补流程，而是本项目防止玩法回归的主要护栏。修改武器、音频、资源、命中、碰撞、模型或 UI 状态时，先判断是否需要补测试，再开始验收。

- 自动测试优先覆盖稳定规则：武器状态、弹匣和换弹、自动/半自动射击、音频缓存、资源存在性、碰撞、评分、模型结构。
- 浏览器验收优先覆盖真实体验：持续射击、鼠标按住连发、控制台错误、Network 请求数量、3D 武器可见性、枪口锚点和 HUD 状态。
- 只跑自动测试不能替代浏览器验收；枪模朝向、模型遮挡、枪口火焰位置、音频请求暴涨这类问题必须在真实页面里看。
- 运行 `npm run test:e2e` 会通过 Playwright/Chromium 进入 `?e2e=1` 验收模式，捕获页面加载期控制台错误，并验证 5 把武器进入靶场后使用 3D 模型。
- `npm run test:e2e` 会输出单武器截图、静态 contact sheet 和开火 contact sheet，用于人工复核 5 把武器（M4、M95、Deagle Golden、AWP、AK47）的屏幕位置和枪口火焰。
- Phase2 TaCZ 静态枪模使用 `npm run test:e2e -- e2e/phase2-tacz-static.spec.js` 单独验收。该测试输出 `phase2-static-*` 截图和姿态报告，不替代普通运行时 `e2e/weapons.spec.js`。
- 如果修复线上或手动复现过的问题，测试里要尽量留下对应的回归用例，避免以后重复踩坑。

## 常见故障排查

### 命中后卡住

优先检查控制台是否出现命中反馈相关错误。飘字、特效、HUD 更新不能中断射击主链路；即使视觉反馈失败，也要保证扣血、扣弹、计分和主循环继续运行。

### 音频请求暴涨

检查 `src/audio.js` 的缓存状态。每个 OGG 应该最多加载一次；加载失败后也要记录失败状态，后续播放 fallback，不能反复 fetch 同一路径。

### 枪口火焰位置错误

先看 `?e2e=1` 的 `snapshot().runtime.muzzleAnchorScreen`、`muzzleFlashScreen` 和 `muzzleFlashDistancePx`。如果火焰在枪身中部，优先调 `modelConfig.muzzleLocalPosition`；不要再添加相机坐标式 `muzzleOffset`。

调试枪火视觉（大小/透明度/旋转）时，通过 `snapshot().muzzleFlash` 读取当前应用配置。运行时可用 `setModelConfig` 临时改 `muzzleLocalPosition`，但 `muzzleFlash` 视觉参数需直接改 `src/config.js` 后刷新页面查看。枪火旋转通过贴图 `wAng` 实现，不再通过 mesh rotation（`BILLBOARDMODE_ALL` 会覆盖后者）。

### 3D 模型异常

先区分模型资源加载失败、Blockbench JSON 解析问题、贴图/材质问题和模型坐标问题。第一人称 2D fallback 已移除，模型异常应通过测试或控制台直接暴露。

如果 e2e 里 `visibleMeshCount` 很高但截图里武器不可辨，优先看 `snapshot().activeModel.screenBounds`：投影宽高、面积比例和中心点才是第一人称视觉是否修好的关键证据。

5 把武器统一走 `loadTaczFirstPersonWeapon()` adapter 路径；adapter 内部创建 TaCZ 原生模型、`FirstPersonRig`、方块手和动画控制器。所有武器按 `FIRST_PERSON_WEAPON_COLORS` 选纯色材质。排查任意武器模型异常时都看 `snapshot().activeModel`（source/partCount/screenBounds/visibleMeshCount）。

#### 原生第一人称定位组排查

当武器整体位置/旋转异常、双手不贴合握把、ADS pose 错位时，优先用 debug snapshot 确认 TaCZ 原生 marker 是否被读取并覆盖到 calibration：

```js
const snapshot = window.__blockTargetRangeDebug.snapshot();
snapshot.activeModel.firstPersonMarkers;
snapshot.activeModel.calibrationSource;
snapshot.activeModel.rigCurrentPose;
snapshot.activeModel.weaponRootPosition;
snapshot.activeModel.weaponRootRotation;
snapshot.activeModel.adsProgress;
```

期望：

- `firstPersonMarkers.idleView/ironView/leftHand/rightHand` 对真实 5 把武器均为 `true`，说明 geo 中四个定位组都被 adapter 读取。
- `calibrationSource` 中对应字段为 `true`，说明 marker 已覆盖 calibration 的 pose/grip。
- `rigCurrentPose` 在腰射时为 `"hip"`，AWP 开镜时随 `adsProgress` 趋近 1 切换到 `"ads"`。
- `adsProgress` 在 AWP 开镜时从 0 平滑趋近 1，关镜或切枪后回到 0。
- `weaponRootPosition` / `weaponRootRotation` 不含 NaN，且与 `hipPose`/`adsPose` 的预期值一致。

若 `firstPersonMarkers` 中某字段为 `false`，先检查对应 geo 文件是否包含该定位组 bone（`idle_view`/`iron_view`/`lefthand_pos`/`righthand_pos`）；若 `calibrationSource` 中某字段为 `false` 但 marker 存在，检查 `mergeCalibrationWithMarkers` 是否被正确调用。

#### V2 新枪 Bedrock geo 的 screenBounds 特性

deagle_golden/m95 走 TaCZ 原生 Bedrock geo renderer（`createTaczGeoModel`），模型包含完整的 bone 层级和所有 cube（弹匣、握把、枪托、子弹等内部部件）。与旧 3 把枪的 Blockbench elements 相比，Bedrock geo 的 cube 数量多（492-764 个 vs 旧枪约 100-200 个），Y/X 跨度更大，导致 `getModelScreenBounds` 的 10/90 百分位投影范围更大。

这直接影响了 E2E 的两个断言：

- `expectUsableScreenProjection`：新枪的 `centerX`/`centerY` 可能偏左上（m95），`areaRatio` 可能偏大。`SCREEN_BOUNDS_OVERRIDES` 为这些枪设置了独立阈值。
- `expectMuzzleNearFrontFor`：枪口的 `relativeY`（枪口在模型投影 Y 范围中的相对位置）可能偏高（deagle_golden=0.82, m95=0.81），因为模型投影包含了弹匣/握把等下方部件。`MUZZLE_FRONT_THRESHOLDS` 为每把新枪设置了独立阈值。

这些阈值差异是模型几何特性，不是枪口位置错误。视觉验收时应确认枪口在枪管前端（截图可见），而非依赖 `relativeY` 数值。

### 构建或测试失败

先运行 `npm run test` 定位逻辑测试，再运行 `npm run build` 定位构建或资源路径问题。完整提交前使用 `npm run check`，视觉相关改动还需要单独运行 `npm run test:e2e`。
