# 项目协作约定

本项目默认使用中文沟通、中文文档和中文注释。新增或修改代码时，优先让后来维护的人能快速读懂玩法规则、资源链路和调试方法。

## 核心原则

- 改动保持小范围，优先沿用现有模块和写法，不顺手做无关重构。
- 代码文件里的复杂逻辑要加必要中文注释，尤其是状态流转、Babylon.js 坐标/模型处理、音频缓存、武器规则、资源 fallback 和性能节流。
- 不需要给简单赋值、普通 import、直观函数堆注释；注释要解释“为什么这样做”或“不这样会出什么问题”。
- 测试是防止玩法回归的核心流程。涉及玩法、资源、音频、3D/2D 显示、UI 交互的改动后，优先补充或更新测试，再运行 `npm run check`。
- 如果因为环境或时间原因没有运行测试，交付说明里必须写清楚未测试的原因和剩余风险。
- 视觉或交互改动需要浏览器验收，重点检查武器朝向、P90 模型、控制台错误、音频网络请求数量和持续射击体验。
- 排查 Babylon.js、MCP、材质、粒子、GUI、模型加载或渲染问题时，先看 `docs/Babylon官方资料与测试方向.md`；本地文档没有合适答案时，要去 Babylon 官方文档、API、Playground、官方论坛或 GitHub 查找，再把可复用结论补回文档。
- 外部资源只复制项目实际使用的最小集合，不整包复制模组；新增资源时在 docs 中记录来源和授权注意事项。

## UI 素材优先规则

- 涉及 HUD、面板、按钮、热栏、准星、图标、进度条、容器背景、武器试验场看板等 UI/GUI 改动时，先到本地素材包查找可复用资源，再决定是否自己设计。
- 素材查找优先级固定为：先查 `我的世界素材/tac-mod-resourcesV2/`，再查 `我的世界素材/tac-mod-resources/`，最后查原版 Minecraft 风格素材目录。
- V2 优先查找路径：`我的世界素材/tac-mod-resourcesV2/assets/tacz/custom/tacz_default_gun/`、`我的世界素材/tac-mod-resourcesV2/assets/tacz/textures/`、`我的世界素材/tac-mod-resourcesV2/assets/tacz/sounds.json`。
- 旧 TaC 备用路径：`我的世界素材/tac-mod-resources/assets/tac/textures/gui/`、`我的世界素材/tac-mod-resources/assets/tac/textures/crosshair/`、`我的世界素材/tac-mod-resources/assets/tac/textures/crosshair_hit/`、`我的世界素材/tac-mod-resources/assets/tac/sounds/`。
- 原版素材最后查找路径：`我的世界素材/gui/`、`我的世界素材/gui/container/`、`我的世界素材/block/`、`我的世界素材/entity/`、`我的世界素材/item/`。
- 如果素材包里有合适资源，只把实际使用的最小文件复制到项目 `assets/` 或 `public/` 下，并在代码配置里引用复制后的稳定路径；不要直接运行时引用 `我的世界素材/`，该目录已被 `.gitignore` 忽略。
- 如果素材包找不到合适资源，再自己设计；自设计 UI 也要保持 Minecraft/像素枪械风格，避免临时灰色矩形、纯 CSS 拼凑感和与素材包风格割裂的面板。
- 新增或替换 UI 素材时，同步更新 `docs/资源与授权.md` 或相关调试文档，说明来源路径、复制目标和用途。

## 测试分层

项目有两层测试，各司其职：

- **Node test**（`npm test`）：覆盖稳定逻辑规则——武器状态、弹匣换弹、自动/半自动、音频缓存、资源存在性、碰撞、评分、模型结构。运行快，开发时频繁跑。
- **Playwright E2E**（`npm run test:e2e`）：浏览器视觉验收——武器切换、3D 模型渲染、控制台错误、截图对比。运行慢（约 40s），提交前或改视觉相关代码后手动跑。
- **不需要 MCP 浏览器**：Playwright headless 比 MCP Chrome 快得多，脚本断言比 AI 看图准。日常验收用 `npm run dev` 手动打开浏览器即可。
- `npm run check` 只跑 Node test + 构建，不包含 E2E，避免拖慢日常开发节奏。CI 或合并前单独跑 E2E。

### E2E 使用强制约束（文本大模型必读）

- **开发阶段禁止跑 `npm run test:e2e`**。E2E 单次运行 6 分钟，失败只报 timeout 不报具体原因，文本模型反复盲猜重跑会大量浪费 token 和时间。
- **开发阶段用 `npm test` + 浏览器 MCP 调试**。需要检查页面状态时，用 Chrome DevTools MCP 的 `evaluate_script` 直接读取 `window.__blockTargetRangeDebug.snapshot()`，秒级反馈具体字段值，不需要等 E2E 超时。
- **E2E 只在交付前跑一次最终验证**，不是调试工具。交付前确认逻辑测试通过 + 浏览器 MCP 检查无异常后，跑一次 E2E 做最终回归。
- **E2E 失败时禁止重跑**。用浏览器 MCP 检查 `selectWeaponAndWaitForModel` 的 5 个条件（currentWeaponId / ready / rootEnabled / visibleMeshCount > 20 / isTaczNative）哪个不满足，针对性修复后再跑。

## 视觉验收分工（强制约束）

**文本大模型不具备视觉能力，禁止参与任何视觉验证工作。** 硬看截图既无法得出正确结论，又大量浪费 token 和时间。以下约束不可违反：

### 文本大模型禁止事项

- **禁止看截图/读图判断视觉问题**：包括但不限于武器朝向、模型位置、UI 布局、准星、枪口火焰、HUD 对齐等。文本模型无法从像素值推断画面表现，硬做只会得出错误结论。
- **禁止从屏幕坐标数值推测"画面应该长什么样"**：E2E 的 screenBounds 断言失败时，文本模型只负责检查代码逻辑（模型是否加载、root 是否启用、mesh 是否可见），不负责判断"武器应该出现在屏幕右侧"这类视觉结论。
- **禁止自行决定视觉验收是否通过**：视觉验收的通过/不通过由多模态大模型判定，文本模型只接收结论和错误描述。

### 文本大模型应该做的

- **完成改动后输出动态验收清单**：列出本次改动涉及的视觉检查项（武器朝向、P90 模型、HUD 布局、准星、控制台错误、音频网络请求数量等，按改动模块动态列）+ 对应 `test-results/` 截图路径 + 每项的期望表现，提示用户切换多模态大模型验证。
- **根据多模态模型的错误描述修复代码**：多模态模型反馈格式必须包含——哪个检查项不通过、截图中具体表现（位置/颜色/朝向偏差）、推测的根因方向（CSS 值、模型旋转、资源路径等）。文本模型将反馈翻译为代码改动，不需要自己"看"。
- **修完重新生成截图交回多模态模型复验**，直到全部通过。

### 多模态大模型职责

- 读图对照验收清单逐项检查，发现问题按格式反馈给文本模型。
- 若当前对话模型本身具备多模态能力，跳过分工，直接自己看图验收。

清单模板、检查项库和反馈格式见 `docs/视觉验收清单.md`。

## 文档同步

- 根目录只保留轻量规则，具体说明写入 `docs/`。
- 调整武器系统、资源路径、调试参数或验收方式时，同步更新对应文档。
- 调整 Babylon MCP、材质、粒子、GUI、模型加载或浏览器视觉验收方向时，同步更新 `docs/Babylon官方资料与测试方向.md`。
- 文档优先使用中文文件名，便于在本项目语境下直接维护。
